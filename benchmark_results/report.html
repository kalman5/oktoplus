1. Speed — Single Client (no pipeline)

Operations per second (1 client, pipeline=1)

Verdict: At parity. All operations within ±8% of Redis. RPOP slightly faster on Oktoplus (+8%), LPOP slightly slower (-6%). I/O round-trip latency is the bottleneck, not CPU.

2. Pipelining (pipeline=16)

Operations per second — Pipeline 16 vs Pipeline 1

Critical Bug: Oktoplus drops to ~391 ops/sec with pipeline=16 — 80x slower than pipeline=1 (31K). This is not “no benefit” — it’s active degradation. Redis reaches 390K+ ops/sec (12x boost). The RESP handler likely synchronizes per-batch instead of per-command.

4. Average Parallelism Scaling

Average ops/sec across all commands vs concurrent clients

Observation: Oktoplus does NOT outscale Redis despite per-key locking. Redis peaks ~89K avg ops/sec, Oktoplus plateaus at ~70K (0.71x at 200 clients). RPUSH at 200 clients has a catastrophic outlier (37K ops/sec, max latency 544ms).

5. Latency at High Concurrency (200 clients)

p99 Latency (ms) — 200 concurrent clients

Tail latency: RPUSH p99=70.9ms (max=544ms!) on Oktoplus vs p99=1.0ms on Redis. All other commands have manageable p99 (~2ms) but RPUSH has a severe contention bug at 200 clients.

6. Recommendations

1. Fix the pipelining bug — #1 priority. Pipelined throughput should be ≥ single-command throughput. Currently 80x slower with pipelining. Likely fix: process multiple RESP commands from a single read() without per-batch synchronization.
2. Fix RPUSH contention at 200 clients — 544ms max latency and 0.33x ratio vs Redis. Specific to RPUSH (LPUSH is fine). Check lock ordering or list tail pointer contention.
3. Profile lock contention — Run perf lock or flamegraph to identify the 10-15% gap at moderate concurrency (10-100 clients).
4. Consider lock-free structures — For read-only ops (LLEN, SCARD), atomic counters could eliminate locking entirely.
