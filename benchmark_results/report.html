Verdict: At parity or better. Oktoplus beats Redis on SCARD (+42%), LLEN (+14%), LPUSH (+6%). LRANGE_100 is the weakest at 0.87x.

2. Pipelining (pipeline=16)

Gap: Oktoplus does not support command pipelining. Redis gets a 10-14x throughput boost (up to 420K ops/sec). This is the single biggest performance gap.

4. Average Parallelism Scaling

Average ops/sec across all commands vs concurrent clients

Surprise: Oktoplus does NOT outscale Redis despite per-key locking. Redis peaks ~86-89K ops/sec, Oktoplus plateaus at ~65-70K (0.79x at 100-200 clients). LPUSH is the worst scaler (0.58x at 200 clients). Likely causes: lock contention, thread-per-connection overhead, or allocator contention.

5. Latency at High Concurrency (200 clients)

p99 Latency (ms) — 200 concurrent clients

Tail latency: LPUSH p99=28.7ms (max=284ms) on Oktoplus vs p99=1.6ms on Redis. RPUSH also elevated at p99=4.0ms (max=29ms). Suggests lock contention storms on list push paths.

6. Recommendations

1. Implement pipelining — 10x throughput sitting on the table. Most Redis clients pipeline by default.
2. Profile LPUSH tail latency — 284ms max at 200 clients is a red flag. Run perf/flamegraph to find the contention hotspot.
3. Try jemalloc/mimalloc — Multiple threads competing for malloc/free may be serializing under load.
4. Async I/O — Replace thread-per-connection with io_context thread pool + async_accept/async_read.
